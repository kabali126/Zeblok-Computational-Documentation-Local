## Overview

This use case walks through an end-to-end workflow in two stages. First, you deploy a large language model as an **Ai-Inference** service with **memory offloading** enabled, so that its key-value (KV) cache spills out of GPU memory onto fast local NVMe storage and host DRAM. Then you deploy **OpenClaw** — a self-hosted AI gateway — as an **Ai-MicroService** and point it at that inference as its model backend. The result is an agentic chat gateway powered by a 70B model that would otherwise not fit comfortably on a single GPU.

Memory offloading here is the same **NVMe / High Bandwidth Flash Memory** capability introduced on the [Spawn Ai-WorkStations](https://zeblokcomputationalprivatelimited.mintlify.app/workstation-and-services/spawn-workstation) page, and the inference itself follows the standard [Spawn Ai-Inference](https://zeblokcomputationalprivatelimited.mintlify.app/workstation-and-services/spawn-inference) flow. This page focuses only on the offloading-specific choices and the OpenClaw connection, so it assumes you are already comfortable with those two pages.

By the end you will have:

- An offloading-enabled **Meta Llama 3.1 70B Instruct FP8** inference, with KV cache offloaded to NVMe and DRAM.
- A **Gen AI API key** to authenticate against the inference endpoint.
- An **OpenClaw** microservice configured to use that inference as its LLM provider, reachable from a browser dashboard.

The pieces connect like this: **you (and any chat channels) → OpenClaw gateway → the offloading inference (vLLM, OpenAI-compatible) on Ai-MicroCloud.**

<Note>
  **Before you start**, make sure you have:

  - An aiDAPTIV™ offloading-capable image tag (such as `offloading-image`) available for your model.
  - A data centre with available **NVMe / High Bandwidth Flash Memory** devices (for example, **Phison labs**).
  - A plan with at least one GPU — a 70B FP8 model needs roughly 77 GB of vRAM.
  - Membership of the namespace you intend to deploy into. If yours is not listed, contact your OrgAdmin.
</Note>

## Part 1 — Deploy the memory-offloaded inference

Navigate to **Workspace → Inference** in the left sidebar. If you have not created any inference services yet, you will see an empty state page. Click the **Start Inference** button in the top-right corner to begin.

<Frame>
  <img src="/images/use-cases/openclaw-memory-offloading/no-inference.png" alt="Inference page showing no inference services created yet" />
</Frame>

<Steps>
  <Step title="Configure Model">
    The **Configure Model** tab defines which model you serve, which runtime image to use, where it runs, and how much memory it can offload.

    ### Model *(required)*

    Use the **Model** dropdown to select the model you want to deploy. For this use case, choose **Meta Llama 3.1 70B Instruct FP8** — a model marked **Public** and **aiDAPTIV™**. The aiDAPTIV™ badge indicates the model is optimised for high-performance inference and can unlock offloading when paired with the right image tag.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/inference-select.png" alt="Model selection dropdown showing Meta Llama 3.1 70B Instruct FP8 and other models, with Public and aiDAPTIV badges" />
    </Frame>

    ***

    ### Image Tag *(required)*

    After selecting the model, choose the **Image Tag**. Select the tag named **offloading-image**, which carries the **Latest** and **aiDAPTIV™** badges. This is the runtime image that enables KV cache offloading to NVMe and DRAM.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/select-offloading-inference-image.png" alt="Image Tag dropdown showing the offloading-image tag marked Latest and aiDAPTIV" />
    </Frame>

    ***

    ### Data Center *(required)*

    Select the **Data Center** where your inference will run. Choose a data centre that has NVMe devices available — in this example, **Phison labs** (Redmond, WA, US), shown as the primary, Pro region.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/select-datacenter.png" alt="Data Center dropdown showing the Phison labs data centre in Redmond, WA, US" />
    </Frame>

    ***

    ### NVMe / High Bandwidth Flash Memory

    Once a data centre is selected, the **NVMe / High Bandwidth Flash Memory** toggle appears. Enabling it attaches a high-speed local flash device to your inference, which is where the offloaded KV cache is stored.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/nvme-bandwidth-flash-memory.png" alt="NVMe / High Bandwidth Flash Memory toggle in the off state, with helper text about single replica and autoscaling" />
    </Frame>

    <Warning>
      Enabling NVMe / High Bandwidth Flash Memory pins your inference to a **single replica and disables autoscaling**, because the offloaded cache lives on one physical device. The **Min Replicas**, **Max Replicas**, and **GPU Threshold** fields no longer drive scaling while this is on.
    </Warning>

    ***

    #### Select an NVMe device

    With the toggle on, a list of available devices is displayed. Each entry shows the host node, device path, serial number, model, mount path, host path, and capacity. Select the device you want to attach.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/select-nvme-device.png" alt="NVMe device list showing available flash devices with device path, serial number, mount path, and capacity" />
    </Frame>

    <Note>
      The list shows **only devices that are available in this data centre**. Devices already in use, or on nodes with no free capacity, are hidden. Take note of the device's **mount path** (for example, `/mnt/nvme1`) — it must match the offload path in the arguments later.
    </Note>

    ***

    ### Plan *(required)*

    Choose a **Plan** with at least one GPU. The form shows a helper line such as *"70B · fp8 needs ~77 GB vRAM. Showing plans with ≥ 1 GPU (this data center has ~96 GB per GPU)."* and filters the list accordingly. Pick a GPU-backed plan such as **g1.Small**.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/select-plan.png" alt="Plan selection dropdown filtered to GPU-backed plans for the 70B FP8 model" />
    </Frame>

    When the Configure Model tab is complete, click **Next** to move to **Configure Environment**.
  </Step>

  <Step title="Configure Environment">
    The **Configure Environment** tab collects the namespace, name, networking, and — most importantly for offloading — the runtime arguments.

    ### Namespace *(required)*

    Select the **Namespace** the inference will be deployed into.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/select-namespace.png" alt="Namespace dropdown showing the available namespaces for the inference" />
    </Frame>

    <Note>
      You must be a member of a namespace to deploy into it. If the namespace you need is not listed, contact your OrgAdmin to be added.
    </Note>

    ***

    ### Inference Name *(required)*

    Enter a unique **Inference Name** to identify the service — for example, `Meta llama 3.1 70b`.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/inference-name.png" alt="Inference Name field with the namespace selected" />
    </Frame>

    ***

    ### Protocol & Ports *(required)*

    A single port is pre-configured by default — **TCP / 8080 / UI-default** — which is the standard vLLM port. Leave it as-is unless you have a reason to change it.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/port-number.png" alt="Protocol and Ports section showing the default TCP port 8080" />
    </Frame>

    ***

    ### Environment Variables *(optional)*

    Add any key-value pairs you need injected into the container at startup. For this use case none are required.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/environment-variable.png" alt="Environment Variables key and value fields, left empty" />
    </Frame>

    ***

    ### Arguments

    This is where offloading is actually configured. When the **offloading-image** is selected, the **Arguments** section arrives pre-filled with the vLLM flags that drive the offloaded run. Review them and adjust only the values you understand.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/add-arguments.png" alt="Arguments list pre-filled with the offloading vLLM flags, including nvme_path and the SSD and DRAM KV offload sizes" />
    </Frame>

    | Argument | Example value | What it controls |
    | --- | --- | --- |
    | `--model` | `RedHatAI/Meta-Llama-3.1-70B-Instruct-FP8` | The model identifier. Derived from your Model selection and locked. |
    | `--nvme_path` | `/mnt/nvme1/inference0` | The path on the attached NVMe device used for offloaded cache. The prefix matches the device's mount path. |
    | `--gpu-memory-utilization` | `0.95` | Fraction of GPU memory vLLM may use before offloading. |
    | `--tensor-parallel-size` | `1` | Number of GPUs to shard across. Derived from the plan and locked. |
    | `--enable-chunked-prefill` | *(flag)* | Splits long prefills into chunks to reduce peak memory. |
    | `--enable-prefix-caching` | *(flag)* | Reuses cached prefixes across requests. |
    | `--enable-auto-tool-choice` | *(flag)* | Enables automatic tool/function calling. |
    | `--tool-call-parser` | `llama3_json` | Parser used for Llama 3 JSON-formatted tool calls. |
    | `--ssd-kv-offload-gb` | `1200` | Amount of KV cache (GB) to offload to the attached NVMe/SSD device. |
    | `--dram-kv-offload-gb` | `64` | Amount of KV cache (GB) to offload to host DRAM (CPU memory). |
    | `--max-model-len` | `40000` | Maximum context length the model will serve. |
    | `--max-num-seqs` | `64` | Maximum number of concurrent sequences. |
    | `--max-num-batched-tokens` | `8192` | Maximum tokens batched per scheduler step. |
    | `--disable-log-requests` | *(flag)* | Suppresses per-request logging. |
    | `--disable-custom-all-reduce` | *(flag)* | Disables the custom all-reduce kernel for compatibility. |

    <Note>
      The three offload-specific flags are `--nvme_path`, `--ssd-kv-offload-gb`, and `--dram-kv-offload-gb`. Together they let the model keep a much larger KV cache than GPU memory alone allows, which is what makes the long `--max-model-len` value practical on a single GPU.
    </Note>

    Once the required fields are filled in, click **Spawn Inference**.
  </Step>

  <Step title="Spawn and open the inference">
    The platform provisions your container and redirects you to the Inference list. Your new service appears with its current status; once it shows **Running**, it is ready.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/spawned-inference.png" alt="Inference list showing the newly created Meta llama 3.1 70b service in a Running state" />
    </Frame>

    Click **View Details** to open the full panel. It shows the endpoints (a public HTTPS URL and a cluster-local URL on port 8080), runtime status, placement, and the configuration — including the **offloading-image** tag and the requested resources.

    <Frame>
      <img src="/images/use-cases/openclaw-memory-offloading/inference-details.png" alt="Inference details panel showing endpoints, runtime, placement, and configuration including the offloading-image tag" />
    </Frame>

    <Note>
      A 70B model can take a few minutes to load before it will answer. The details panel and chat will indicate when it is online.
    </Note>
  </Step>
</Steps>

## Important Rules

<Warning>
  Read these before running this workflow in a shared or production environment.
</Warning>

- **Gen AI keys are shown only once.** Copy or download the key the moment it is created and store it securely. A lost key cannot be recovered — you must create a new one.
- **Redact everything sensitive in screenshots:** API keys, NVMe device serial numbers, public endpoints, gateway tokens, dashboard passwords, namespaces, and Kubernetes names.
- **Offloading means a single replica.** Enabling NVMe / High Bandwidth Flash Memory disables autoscaling for the inference, because the offloaded cache is tied to one physical device.
- **NVMe devices are data-centre-specific.** Only available devices in the selected data centre are shown; in-use devices and full nodes are hidden. The device's mount path must match the `--nvme_path` prefix in the arguments.
- **Treat the offloading arguments with care.** `--model` and `--tensor-parallel-size` are derived and locked. The offload sizes (`--ssd-kv-offload-gb`, `--dram-kv-offload-gb`) and `--max-model-len` are what let the model serve long contexts — change them only if you understand the memory trade-offs.
- **Always use the public HTTPS endpoint** (suffixed with `/v1`) as the OpenClaw Base URL. The cluster-local URL will not work as an external gateway target.
- **The setup port (8090) is for one-time configuration.** Once OpenClaw is launched, day-to-day use happens on the control dashboard (port 18789).
