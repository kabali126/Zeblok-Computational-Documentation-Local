## Overview

The **API Keys & Secrets** page is accessible via **Platform → API Keys & Secrets** in the left sidebar. It is where you generate and manage every credential used for programmatic access to the platform — from your terminal, your CI/CD pipelines, your SDK code, and your AI agents.

The page is organised into three tabs, each managing a different kind of credential:

| Tab | Credential | Used for |
| --- | --- | --- |
| **Microcloud** | An API Key + API Secret pair | Programmatic access to the Ai-MicroCloud platform API. |
| **Object Store** | Endpoint, user, and secret key | Access to the MinIO / S3-compatible object storage console and buckets. |
| **Gen AI** | Named API keys (multiple) | Authenticating Gen AI integrations and AI agent accounts. |

Every credential on this page authenticates **as the user who created it** — inheriting that user's privilege role, IAM roles, and effective permissions. A key is not a separate identity with its own permission set; it is a programmatic stand-in for you. Credentials are also organisation-scoped: a key generated by a user in one organisation can only ever access that organisation's resources.

### Who can manage credentials

| Role | Scope |
| --- | --- |
| **SuperAdmin** | Full management of API keys and secrets across every organisation. |
| **Admin** | Full management of API keys and secrets within their own organisation. |
| **Standard User** | Can generate and manage **their own** API keys and secrets. This is one of the few IAM-adjacent capabilities a Standard User has. |

<Warning>
The credentials on this page are live secrets. Treat them the way you would treat a password:

- **Never commit them to source control.** Use environment variables, a secrets manager, or your CI/CD provider's encrypted-secrets feature.
- **Most secret values are shown only once.** If you lose a value, your only option is to regenerate — which invalidates the old credential.
- **Anyone holding your key can act as you.** A leaked key has every permission your account has, scoped to your organisation.
</Warning>

## Microcloud Credentials

The **Microcloud** tab manages a single API Key and API Secret pair for programmatic access to the platform API.

![API Keys & Secrets page on the Microcloud tab showing the empty state with a Generate Keys button](/images/iam/api-keys/api-keys-secrets.png)

### Generating your key pair

If you have not generated credentials yet, the tab shows an empty state. Click **Generate Keys** to create your API Key and API Secret pair.

When the pair is created, a dialog displays both values and the credentials are **auto-downloaded** to your machine as a `.env` file (named like `zeblok-microcloud-2026-05-14.env`). The dialog warns you to store the credentials securely — if you lose them, your only recovery path is to regenerate, which invalidates the current pair.

![Microcloud Credentials dialog showing the generated API Key and API Secret with a security warning, values redacted](/images/iam/api-keys/generate-keys.png)

After the dialog is closed, the tab shows your key pair in a persistent panel: the API Key with a partially-masked value you can copy, the API Secret fully masked, and the date the pair was last rotated.

### Regenerating your key pair

There is no way to add a second Microcloud key pair — each account has exactly one. To replace a compromised or lost pair, click **Regenerate Keys**.

![Regenerate Microcloud Credentials confirmation dialog warning that the current key and secret will be permanently invalidated](/images/iam/api-keys/regenerate-keys.png)

<Warning>
Regenerating is immediate and irreversible. The confirmation dialog states it plainly: *"This will permanently invalidate your current API key and secret. Any integrations using the old credentials will stop working immediately."* Before regenerating, make sure you have a plan to update every script, pipeline, and integration that uses the old pair.
</Warning>

## Object Store Credentials

The **Object Store** tab displays the credentials for the platform's MinIO / S3-compatible object storage — used to access storage buckets and the MinIO console.

![Object Store tab showing the Bucket Access Endpoint, Console URL, User, and Secret Key fields with masked values](/images/iam/api-keys/object-store-credentials.png)

Unlike the Microcloud tab, there is nothing to generate here — these credentials are provisioned for you and are used to sign in to the MinIO console. The tab shows four values, partially masked:

- **Bucket Access Endpoint** — the S3-compatible API endpoint your code or S3 client connects to.
- **Console URL** — the web URL for the MinIO console UI.
- **User** — the object-store username.
- **Secret Key** — the object-store secret, masked by default.

Click **View Credentials** to open a dialog showing all four values in full, each with a copy button, plus a **Download** button to save them.

![Object Store Credentials dialog showing the full Bucket Access Endpoint, Console URL, User, and Secret Key values, secret redacted](/images/iam/api-keys/view-object-store-credentials.png)

<Note>
The Object Store Secret Key cannot be rotated or regenerated from this page — it is fixed for the credential set provisioned to your account. Because it cannot be rotated, be especially careful never to expose it. If you believe it has been compromised, contact your platform administrator.
</Note>

## Gen AI Credentials

The **Gen AI** tab is a full multi-key manager for credentials used by Gen AI integrations and AI agents. Unlike the single Microcloud pair, you can create multiple Gen AI keys — **each agent can hold up to 5 active keys.**

![Gen AI tab showing the empty state with a Create Key button](/images/iam/api-keys/gen-ai-credentials.png)

### Creating a Gen AI key

Click **+ Create Key** to open the **Create Gen AI Key** dialog. The dialog reminds you up front that *the secret is shown only once after creation.*

![Create Gen AI Key dialog showing the Name, Type, and Expiry fields](/images/iam/api-keys/create-gen-ai-key.png)

Fill in the following fields:

- **Name** *(required)* — a descriptive label for the key, for example `Production Agent Key`. The name appears in the key list so you can identify which integration a key belongs to.

- **Type** *(required)* — choose how the key will be used:
  - **User — personal API key** — a key used directly by you, the human user, for personal Gen AI API access.
  - **User Agent — key for an AI agent account** — a key intended to be held and used by an AI agent rather than a person.

  ![Type dropdown showing the User personal API key and User Agent options](/images/iam/api-keys/gen-ai-key-type.png)

- **Expiry** *(required)* — how long the key remains valid. Choose **7 days**, **30 days**, **90 days**, or **Custom** to set your own period.

  ![Expiry dropdown showing the 7 days, 30 days, 90 days, and Custom options](/images/iam/api-keys/gen-ai-key-expiry.png)

Click **Create Key**.

### Copying the new key

On creation, a dialog displays the full key value with a clear warning: *"Copy this key now — it will not be shown again. If lost, create a new key."* Copy the value and store it securely, or use the **Download** button to save it. Once you close this dialog, the full value is gone — the key list only ever shows a truncated Key ID afterward.

![Gen AI Key Created dialog showing the full API key value (redacted) with a copy-now warning and Download button](/images/iam/api-keys/after-gen-ai-key-generated.png)

The new key now appears in the Gen AI Credentials list with its Name, Type, truncated Key ID, Created date, Expiry date, and Status.

### Key expiry

When a Gen AI key reaches its expiry date, it **becomes invalid and stops working** — any integration using it will start failing authentication. Plan ahead: before a key expires, create a replacement and update your integrations, then revoke the old key.

### Revoking a Gen AI key

To remove a key before its expiry, click the delete (trash) icon in the **Actions** column of the key's row. A confirmation dialog appears.

![Revoke GenAI Key confirmation dialog warning that the key will be permanently deactivated and services using it will lose access](/images/iam/api-keys/delete-gen-ai-key.png)

<Warning>
Revoking is permanent and immediate. The confirmation dialog states: *"This key will be permanently deactivated. Any services using it will immediately lose access."* There is no undo and no way to reactivate a revoked key — if you revoke a key still in use, you will need to create a new one and update every integration that depended on the old one.
</Warning>

## Important Rules

A few points apply across every credential on this page:

- **Credentials act as you.** Every key and secret inherits the permissions of the user who created it. There is no separate permission model for keys — a key can do exactly what its creator can do, no more and no less.
- **Credentials are organisation-scoped.** A key created by a user in one organisation can only access that organisation's resources. Organisation isolation applies to programmatic access exactly as it does to the UI.
- **Secret values are shown once.** Microcloud secrets, Gen AI key values, and the Object Store secret are displayed in full only at creation (or, for Object Store, via View Credentials). Treat the moment of creation as your one chance to capture the value.
- **Regenerating and revoking break live integrations.** Regenerating a Microcloud pair or revoking a Gen AI key takes effect immediately, with no grace period. Always update dependent integrations as part of the same change.
- **Standard Users manage their own credentials.** Even though Standard Users have no broader IAM management access, they can generate and manage their own API keys and secrets — this is a deliberate exception so that every user can integrate programmatically.
