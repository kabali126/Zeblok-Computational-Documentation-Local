## Overview

The **Data Centers** page is accessible via **Platform → Configurations → Data Center** in the left sidebar. It gives you a complete view of all clusters provisioned for your organisation, along with their status, location, and underlying node infrastructure.

## Primary Cluster

Every Ai-MicroCloud deployment has exactly **one Primary cluster**. This is the central hub where the core Ai-MicroCloud platform runs — hosting the control plane, the management dashboard, workloads, and all platform services. All other resources, users, and secondary data centres are managed and orchestrated from this primary cluster.

The primary cluster is always listed at the top of the Data Centers page under the **Primary clusters** section. Each primary cluster card displays the following information at a glance:

- **Name** — the identifier of the cluster (for example, Phison-Prod)
- **Status** — whether the cluster is currently Running or not
- **Location** — the physical region of the cluster (for example, Redmond, US)
- **Category** — the cluster category, such as non-edge
- **Cluster type** — Primary
- **DNS Prefix** — the DNS prefix used for services deployed in this cluster
- **Bandwidth** — the network bandwidth available (for example, 1 gbps)

## Secondary Data Centers

In addition to the primary cluster, your organisation can connect one or more **Secondary data centers** to the Ai-MicroCloud platform. Secondary data centers extend the reach of your primary cluster — allowing you to run workloads closer to your users, meet data residency requirements, or leverage additional compute capacity from different infrastructure providers.

Secondary data centers are joined to the primary cluster and are fully managed from the same Ai-MicroCloud dashboard. Once connected, they appear in the Data Centers list alongside the primary cluster and can be selected when spawning workstations, microservices, or inference services.

Secondary data centers can be hosted on any of the following:

- **AWS** — Amazon Web Services cloud infrastructure, ideal for organisations already running workloads on AWS or needing access to AWS regions worldwide.
- **Azure** — Microsoft Azure cloud infrastructure, suitable for organisations with existing Azure footprints or those requiring Azure-specific compliance and regional coverage.
- **On-Premise** — Your organisation's own physical hardware, deployed in your own data centre facility. This is ideal for organisations with strict data sovereignty requirements, air-gapped environments, or those who want full control over their infrastructure.

Regardless of where a secondary data center is hosted, it is centrally managed from the primary Ai-MicroCloud cluster. Users interact with all data centers — primary and secondary — through the same unified interface.

<Note>
  Contact your platform administrator to connect a new secondary data center to your Ai-MicroCloud deployment. Once joined, it will appear automatically in the Data Centers list and become available for workload deployment.
</Note>

## Cluster Options

Hovering over a cluster card reveals a context menu with three actions:

- **Details** — opens a modal with the full configuration of the cluster
- **Nodes** — navigates to the node list for the cluster
- **Delete** — removes the cluster from your organisation

## Cluster Details

Clicking **Details** opens a modal showing the complete configuration of the selected cluster. The details panel includes:

| Field | Description |
|---|---|
| **Category** | The cluster category — Non-Edge or Edge |
| **Storage Classes** | The storage classes available on the cluster, such as Hub CSI and Edge CSI-E |
| **Cluster Type** | Whether this is a Primary or Edge cluster |
| **Visibility** | Whether the cluster is Public or Private |
| **Location** | The full physical address of the data centre |
| **DNS Prefix** | The DNS prefix used for all services deployed in this cluster |
| **Bandwidth** | The network bandwidth of the cluster |
| **Docker Registry URL** | The URL of the Docker registry configured for the cluster, if any |

## Nodes

Clicking **Nodes** from the cluster card or the context menu navigates to the node list for that cluster. Nodes are divided into two sections:

### Control Plane

Control plane nodes manage the Kubernetes cluster itself. They handle scheduling, API requests, and cluster state. Each control plane node card displays:

- **Node name** and **IP address**
- **Status** — Running or otherwise
- **Role** — labeled as `control-plane`
- **CPU**, **RAM**, and **Pod** capacity
- **Kubernetes version** and **architecture**
- **Uptime** since last restart

### Worker Nodes

Worker nodes are where your actual workloads — workstations, microservices, and inference services — run. Each worker node card displays:

- **Node name** and **IP address**
- **Status** — Running or otherwise
- **Role** — labeled as `worker`
- **CPU**, **RAM**, and **GPU** capacity
- **GPU model** — for example, NVIDIA RTX 6000 Ada Generation
- **Kubernetes version** and **architecture**
- **Uptime** since last restart

## Node Monitoring

Each node has a **Monitor node** button (shown as a graph icon on the node card). Clicking it opens a live monitoring panel on the right side of the screen for that specific node.

<Frame>
  <img src="/images/datacenter/datacenter-node-monitoring-click.png" alt="Monitor node button highlighted on a control plane node card" />
</Frame>

The **Node Monitoring** panel shows real-time metrics for the selected node, including:

- **CPU Usage** — current CPU utilisation as a percentage of total cores
- **Memory Usage** — current RAM utilisation
- **Network RX** — incoming network traffic in MB/s
- **Network TX** — outgoing network traffic in MB/s

It also includes a **Usage History** section with time-series charts for CPU, Memory, and Network RX metrics. You can filter the history by time range — **5 minutes**, **1 hour**, or **24 hours**.

<Frame>
  <img src="/images/datacenter/datacenter-node-monitoring.png" alt="Node Monitoring panel showing live CPU usage, memory, network RX and TX metrics with usage history charts" />
</Frame>

<Note>
  Node monitoring data is live — the panel updates in real time as long as it remains open. The status indicator in the top-right of the panel shows **Live** when the connection is active.
</Note>
